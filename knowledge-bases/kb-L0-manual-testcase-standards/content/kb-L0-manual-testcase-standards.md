# Manual Test Case Best Practices — Knowledge Base
### kb-L0-manual-testcase-standards v1.0.0
### Standards for creating manual test cases in XRay. All test agents MUST comply.

---

## MT1: Test Case Structure (XRay Format)

```
Test Key: TC-{auto-generated by XRay}
Summary: One-line description matching scenario title
Priority: Critical | High | Medium | Low
Component: {bounded context — Applications, Loans, Payments, BackOffice, UI}
Labels: [regression, smoke, {epic-id}, {product-type}]
Linked Story: US-XXX
Linked Scenario: TS-XXX

Preconditions:
  - Environment: {dev | staging | prod}
  - User role: {loan applicant | operations agent | system admin}
  - Data setup: {what must exist before test starts}

Test Steps:
  | # | Action | Data | Expected Result |
  | 1 | Navigate to / perform action | Input values | Observable outcome |
  | 2 | ... | ... | ... |

Test Data:
  - Specific values to use (no PII — use synthetic data)
  - Boundary values where applicable

Postconditions:
  - State after test completes
  - Cleanup required (if any)
```

## MT2: Step Writing Rules

1. Each step is ONE atomic user action — not "fill in the form" but "enter SAR 50,000 in Amount field"
2. Expected result on EVERY step — not just the final step
3. Steps must be reproducible by any tester without domain knowledge
4. Include exact values, not "enter a valid amount" — say "enter SAR 50,000"
5. Include navigation: "Click the 'Submit' button" not "submit the form"
6. For API tests: include exact endpoint, method, headers, request body
7. For UI tests: include exact screen, field, button names as they appear

## MT3: Test Data Standards

1. NEVER use real PII — always synthetic data
2. Standard test personas:

| Persona | Name | National ID | Income | Employer | Purpose |
|---|---|---|---|---|---|
| Standard applicant | Ahmed Test | 1234567890 | SAR 15,000 | Test Corp | Happy path |
| Low income | Fatima Test | 2345678901 | SAR 5,000 | Test Ltd | SIMAH rejection |
| High income | Khalid Test | 3456789012 | SAR 40,000 | Test Gov | Cap testing |
| CITC fail | Sara Test | 4567890123 | SAR 12,000 | FAIL_CORP | CITC rejection |
| Boundary min | Omar Test | 5678901234 | SAR 4,000 | Test Min | Min amount boundary |

3. Test card numbers: `4111111111111234` (valid), `4111111111110000` (decline)
4. Dates: use relative dates ("today + 30 days") not absolute dates

## MT4: XRay-Specific Rules

1. Test cases linked to stories via XRay "Tests" link type
2. Test sets grouped by epic: "EP-01 Test Set", "EP-02 Test Set"
3. Test plans per sprint: "Sprint 1 Test Plan" containing relevant test sets
4. Test executions tracked per environment (dev, staging, prod)
5. Defects linked to failed test executions
6. Labels for filtering: `regression`, `smoke`, `sanity`, `e2e`, `{epic-id}`

## MT5: Coverage Matrix

Every test case MUST map to:
```
Story → AC → Scenario → Test Case
US-004 → AC#1 → TS-EP01-US004-01 → TC-XXX
```

XRay test coverage report must show:
- % of stories with test cases
- % of ACs with test cases
- Unlinked test cases (orphans)
- Stories without test cases (gaps)

## MT6: Regression Test Pack

1. Tag regression tests with label `regression`
2. Regression tests are: all brownfield regression scenarios + all happy path scenarios
3. Smoke tests (subset): one happy path per epic, tagged `smoke`
4. Sanity tests (subset): critical path only (create → assess → offer → accept → disburse), tagged `sanity`
5. Regression pack executed: every deployment to staging and prod
6. Smoke pack executed: every deployment to dev

## MT7: Test Case Quality Checklist

- [ ] Summary is clear and unique (no duplicates)
- [ ] Priority assigned and justified
- [ ] Preconditions are complete (a tester can set up without asking)
- [ ] Every step has an expected result
- [ ] Test data uses synthetic personas (no PII)
- [ ] Linked to story and scenario
- [ ] Labels include `regression` if applicable
- [ ] Postconditions describe cleanup if needed
- [ ] Steps are atomic (one action per step)
- [ ] Boundary values tested where applicable

## MT8: Worked Example — XRay Test Case for TS-EP01-US004-02

```
Summary: Verify DE caps BridgeNow amount at SAR 30,000
Priority: Critical
Component: Applications
Labels: [regression, smoke, EP-01, BridgeNow]
Linked Story: US-004
Linked Scenario: TS-EP01-US004-02

Preconditions:
  - Environment: staging
  - User role: system administrator
  - Data setup: BridgeNow product type registered (US-001 complete)
  - Feature flag bridgenow.product.enabled = ON

Test Steps:
  | # | Action | Data | Expected Result |
  | 1 | Create BridgeNow application via API: POST /api/v1/applications | productId: BridgeNow ID, monthlySalary: 40000, requestedAmount: 50000, requestedTenure: 24, fullName: "Khalid Test", nationalId: "3456789012", employer: "Test Gov" | 201 Created, status: "Draft" |
  | 2 | Submit application: POST /api/v1/applications/{id}/submit | Application ID from step 1 | 200 OK, status: "Submitted" |
  | 3 | Run assessment: POST /api/v1/applications/{id}/assess | Application ID from step 1 | 200 OK, status: "Approved" |
  | 4 | Verify approved amount | Check response body field "approvedAmount" | approvedAmount = 30000 (NOT 40000 — capped at SAR 30,000 despite 1x income = 40,000) |
  | 5 | Verify assessment audit log: GET /api/v1/audit | Filter by application ID | Audit entry exists with action "ASSESSMENT_PASSED", amount 30000 |

Test Data:
  - Persona: High income (SAR 40,000) — triggers cap
  - Expected: 1x income = 40,000 but cap = 30,000, so approved = 30,000

Postconditions:
  - Application in Approved status with approvedAmount = 30,000
  - No cleanup required (in-memory DB resets)
```

## MT9: XRay Integration Checklist

When creating test cases in XRay, verify:
- [ ] Test case linked to story via "Tests" relationship
- [ ] Test case added to epic test set ("{epic-id} Test Set")
- [ ] Test set added to sprint test plan
- [ ] Labels match: regression/smoke + epic-id + component
- [ ] Test execution environment specified (dev/staging/prod)
- [ ] Defect template ready (linked to test execution on failure)
